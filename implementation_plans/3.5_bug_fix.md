# 3.5 — Destination not-found menu + rename intent + file/folder-aware templates (Bugs 7, 1 & 2)

Fix three bugs: the silent current-dir fallback when a folder isn't found (Bug 7), mash treating the new filename in rename prompts as a destination folder (Bug 1), and every delete/copy using the folder-safe flag regardless of whether the source is a file (Bug 2).

**Goal:** When destination folder is not found the user sees an explicit menu (create / use current dir / retype). When rename intent is detected, the "to X" token is treated as the new filename — never as a folder — and the rename template uses `before`/`after` placeholders so the LLM cannot confuse it with a directory move. For delete and copy, the LLM receives the resolved source type (file or directory) and picks the correct template variant, preventing `rm -rf` and `cp -r` from firing on plain files.

## Phase 7 — `--dry-run` and `--yes` expected output reference

`--dry-run` and `--yes` bypass interactive menus intentionally — this is by design for faster testing, not a bug to fix. This phase documents exactly what these flags produce so that future test runs and Claude agents can distinguish expected auto-select output from genuine failures.

**`--dry-run` contract:**

| Scenario | Expected output |
|----------|----------------|
| Multiple source candidates | `[dry-run] Auto-selected: <shallowest path>` — no menu shown |
| Single source candidate | `[dry-run] Auto-selected: <path>` — no confirmation prompt |
| No source candidate | No auto-select line — falls through to LLM with no resolved path |
| Multiple destination candidates | `[dry-run] Auto-selected destination: <shallowest path>` — no menu |
| No destination candidates | `[dry-run] Auto-selected destination: .` — defaults to current dir, no menu |
| Filename format menu | `[dry-run] Auto-selected filename: <format 1>` — no menu |
| Command execution | `[dry-run] command not executed.` printed after `Run:` — command never runs |

**`--yes` contract:**

| Scenario | Expected output |
|----------|----------------|
| Multiple source candidates | `Auto-selected: <shallowest path>` — no menu shown |
| Single source candidate | No print — accepted silently |
| Multiple destination candidates | `Auto-selected destination: <shallowest path>` — no menu |
| No destination candidates | Defaults to `.` silently — no menu |
| Filename format menu | `Auto-selected filename: <format 1>` — no menu |
| Command execution | Command runs immediately — no `[y/N]` prompt |

## Claude Instructions

- **Architecture (Bug 7):** The destination-resolution block is in `run()` around line 260. `dest_token` is already extracted before `run()` enters the resolver. The `create_destination` flag is a new local boolean, defaulting `False`. Pass it out of the resolution block alongside `destination`.
- **`_apply_destination` scope:** It currently does `parts[-1] = destination.rstrip("/") + "/"` on `mv`/`cp` commands. For compound `cmd` strings containing `&&`, apply the rewrite only to the sub-command after the last `&&` to avoid corrupting the `mkdir -p` segment.
- **Blank-input semantics for the not-found menu:** blank = cancel. This replaces the old 3.4 no-match flow (blank = accept current) entirely.
- **`--yes` / `--dry-run` (Bug 7):** Both skip interactive prompts. For no-match destination they resolve to `.` without creating anything. Never auto-create a directory — that is a destructive action the user has not confirmed.
- **Architecture (Bug 1):** `_detect_rename_intent` runs before `_detect_destination_idx`. When it returns `True`, skip all of `dest_token` / `resolve_dirs` / destination menu. The "to X" token becomes `rename_target` instead. `ask_llm` gets a new `after: str | None` param; when set it appends `Resolved after: <name>` to the LLM prompt. The rename template in `templates.py` changes to `mv {before} {after}` so the LLM uses the right placeholder names. `_apply_rename` post-processes the LLM output to enforce the `after` filename as the last `mv` argument, preventing the LLM from silently emitting a directory path.
- **`move … and rename it to` edge case:** Both move intent and rename intent are present. Detection order: if "rename" (or synonym) appears anywhere in the prompt AND a "to X" marker follows it, treat X as `rename_target` and the earlier "to Y" (if any) as the destination folder. The LLM receives both `Resolved destination` and `Resolved after` and constructs `mv ./src ./dest/after`.
- **Order:** Implement Phase 3 (rename) after Phase 1–2 (not-found menu) are working, since they touch overlapping destination-detection logic.
- **Architecture (Bug 2):** `source_type` is determined immediately after `resolved` is set, using `os.path.isfile` / `os.path.isdir`. It is passed into `ask_llm` alongside `resolved`. The post-processing safety net (strip `-rf` / `-r` for files) runs after `ask_llm` returns, before `_apply_source` / `_apply_destination`. Templates split into `_file` / `_folder` variants; the formatted string passed to the LLM lists all variants so it can pick the right one given the source type hint.
- **Unknown type fallback (Bug 2):** When `source_type == "unknown"` (no resolved path, ghost file), the LLM receives no `Source type:` line and is told to prefer the file-safe variant. The post-processing net mirrors this and DOES strip `-rf`/`-r` for unknown types, so `delete ghost.xyz` always lands on `rm ./ghost.xyz`. Only `source_type == "directory"` preserves recursive flags.
- **Order:** Phase 5 (file/folder templates) is independent of Phases 1–3 and can be implemented in any order, but should land after the template string is stable (i.e. after the rename template change in Phase 3).
- **3.4 prompt-wording drift (Bug 6):** Phase 1 in this plan replaces the old 3.4 no-match destination flow ("blank accepts current") with the new not-found menu (blank cancels). The multi-match source menu from 3.4 already treats blank as cancel, so no change there — but audit `disambiguate()` and the destination multi-match menu wording to make sure no surviving prompt still claims "Enter to accept" while behaviour is now cancel. Update prompt text where stale.
- **Architecture (Bug 6):** Every `input()` call must be wrapped in `try/except EOFError`. Blank-after-strip and EOFError are treated identically — both cancel. The fix must cover: `disambiguate()`, the source no-match retype prompt, the destination not-found menu, the filename format prompt, the extension submenu, and `confirm_and_run`'s `[y/N]` prompt. In `confirm_and_run`, EOF on `[y/N]` means "no" (do not run), not a crash.
- **`--dry-run` / `--yes` (Phase 7):** These flags bypass menus intentionally. Do not alter their auto-select behaviour. Phase 7 is documentation only — no code changes. If observed output differs from the Phase 7 table, update the table to match reality and note any discrepancy.
- **Constraints:** Do not touch `_apply_source`, `_apply_filename`, or the source/filename menu flows.
- **Testing:** Run all Phase 4, 5, and 6 verifications inside `Claude_testing/` using `--dry-run` and `printf` for piped input. Phase 7 verification uses both flags and compares against the contract tables.
