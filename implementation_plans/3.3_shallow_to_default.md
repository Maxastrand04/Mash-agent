# 3.3 — Shallowest path as default (breadth-first candidate ordering)

When multiple paths match the same stem, auto-select and menu position should prefer the shallowest one. Currently `find` output is depth-first, so `./docs/main.py` appears before `./main.py` — making the subdirectory file the auto-selected candidate even when the root-level file is almost certainly what the user means.

**Goal:** Candidates returned by `resolve_paths` and `resolve_dirs` are sorted by path depth (fewest components first). The shallowest match is `candidates[0]`, so `--yes` / `--dry-run` auto-select and the top menu position both default to it.

**Status legend:** ⬜ Not started · 🟡 In progress · ✅ Done

---

## Bug evidence

```
$ mash --dry-run rename main.py to app.py

Mash found multiple candidate files:
  1. ./docs/main.py      ← deeper, listed first due to find DFS order
  2. ./main.py           ← shallower, listed second

[dry-run] Auto-selected: ./docs/main.py   ← wrong default
Run: mv ./docs/main.py ./app.py
```

Expected default: `./main.py` (depth 1) over `./docs/main.py` (depth 2).

---

## Root cause

`find .` emits paths depth-first. `resolve_paths` iterates that output in order, so the `stems` dict preserves DFS insertion order. The first path added for a given stem is the deepest one encountered before siblings at the root level.

Example tree output order from `find`:
```
.
./docs
./docs/main.py     ← inserted into stems["main"] first
./main.py          ← inserted second
```

`disambiguate` always auto-selects `candidates[0]` and places it at menu position 1. Since candidates follow DFS insertion order, the deepest file wins by default.

---

## Phase 1 — Sort by depth in resolver.py

**File:** `mash/resolver.py`

After building `result`, sort by the number of path components (depth). Use `p.count("/")` as the depth key — shallow paths have fewer slashes.

```python
result.sort(key=lambda p: p.count("/"))
```

Apply the same sort to `resolve_dirs`.

The sort is stable so paths at equal depth keep their original relative order (alphabetical within a depth level, since `find` output is alphabetically ordered within each directory).

## Claude Instructions

- **Scope:** Only `mash/resolver.py`. No changes to `main.py` or any other file.
- **Sort key:** `p.count("/")` counts path separators. `./main.py` has 1 slash, `./docs/main.py` has 2. Sorts ascending so shallower paths come first.
- **Stability:** Python's `list.sort()` is stable. Paths at the same depth keep their relative insertion order (DFS alphabetical within a directory).
- **Both functions:** apply the same sort to `resolve_dirs` — destination directories should also default to the shallowest match.
- **No change to cutoff or n:** this is purely an ordering change. Candidates returned are the same set, just re-ordered.
