# 4.0 — OOP Refactor

Restructure `mash/main.py` into a modular package with `helpers/` (pure logic + I/O), `flows/` (per-intent orchestration), and a thin `main.py` delegator.

**Goal:** Replace the 1466-line monolith with a clean module layout where intent parsing, I/O, LLM calls, command sanitization, and file selection are independently importable and testable. Behaviour must be identical to the current implementation after the rewrite. Update `CONTEXT.md` to reflect the new architecture.

**Status legend:**  ⬜ Not started · 🟡 In progress · ✅ Done

---

## Target directory structure

```
mash/
  __init__.py
  main.py                          — parse args, parse intent, dispatch to flow, confirm_and_run
  intent.py                        — Intent dataclass + parse_intent() factory (pure)
  helpers/
    __init__.py
    console.py                     — Console class (thin I/O: menus, prompts, yes/dry_run)
    llm.py                         — LLMClient class (model, SYSTEM_PROMPT, ask)
    commands/
      __init__.py                  — re-exports from submodules
      sanitize.py                  — pure command transforms
      templates.py                 — hardcoded bash templates (moved from mash/)
      reference.py                 — shell command reference list (was mash/commands.py)
    files/
      __init__.py                  — re-exports from submodules
      extension.py                 — EXTENSION_MAP, reconcile, collect, format (pure)
      resolver.py                  — resolve_paths, resolve_dirs (moved from mash/, unchanged)
      context.py                   — get_directory_context()
    selection/
      __init__.py                  — re-exports select_source, select_destination
      source.py                    — select_source (uses Console + resolver)
      destination.py               — select_destination (uses Console + resolver)
  flows/
    __init__.py                    — re-exports all flow functions
    move_copy.py                   — move/copy flow
    rename.py                      — rename flow
    delete.py                      — delete flow
    create.py                      — create flow
    list.py                        — list flow + scoped_context, list_entries, scope_label helpers
    open_cat.py                    — open/cat flow
```

## Claude Instructions

- **Program flow:** `main.py` parses args and intent → dispatches to the correct flow → flow orchestrates its lifecycle (source/dest selection, LLM call, post-processing) and returns a `str | None` (the final shell command) → `main.py` calls `console.confirm_and_run(cmd)`. Clean entry, clean exit.
- **Architecture layers:** `Intent` (pure data) → `Console` (I/O boundary) + `LLMClient` (LLM boundary) → `helpers/` (pure logic + selection business logic) → `flows/` (per-intent orchestration) → `main.py` (delegator). No module should import from `flows/` or `main.py` except `main.py` itself.
- **Import conventions:** Sub-packages (`commands/`, `files/`, `selection/`) re-export through `__init__.py`. Consumers import from the package, not individual files (e.g. `from mash.helpers.commands import sanitize` not `from mash.helpers.commands.sanitize import apply_source`).
- **`yes`/`dry_run` ownership:** These flags live on `Console` only. No flow or helper function accepts them as parameters — they read from the `Console` instance.
- **`Intent` purity:** `parse_intent` extracts what's knowable from args alone. If ambiguity requires user prompts (e.g. file-vs-folder for bare `create`), the flow resolves it — not the parser. `is_create_file` / `is_create_folder` may both be `False` when ambiguous.
- **`Console` thinness:** `Console` owns raw I/O primitives (render menu, get input, confirm y/n, print). Business logic (candidate ranking, path scoring, "create new folder?" decisions) lives in `selection/` or flows, not on `Console`.
- **`extension.py` purity:** No I/O in this module. Interactive extension picking (`confirm_extension_change`, `_pick_extension`, `_pick_extension_auto`) moves into the flows that need them (`create.py`, `rename.py`). Those flows call `Console` for I/O and `extension.py` for logic.
- **`sanitize.py` role:** Holds pure transform functions. Flows decide which transforms to apply and in what order. `create_flow` builds its own `touch`/`mkdir -p` command directly.
- **`resolver.py`:** Public API unchanged (`resolve_paths`, `resolve_dirs`). Only the file location changes.
- **`commands.py` rename:** The old `mash/commands.py` becomes `helpers/commands/reference.py` to avoid `commands/commands.py` stutter.
- **`LLMClient`:** Class with constructor `model` parameter (default `"qwen2.5-coder:7b"`). `SYSTEM_PROMPT` is a class constant. Designed for future model swapping.
- **Conventions:** Keep the same function/method names where possible. Private helpers inside a class stay `_`-prefixed. Do not rename the public API of `resolver.py`.
- **Constraints:** Do not add new features, error handling, or abstractions beyond the refactor. Do not change any prompt text, menu wording, or user-visible output. Do not use `.pth` files (see CLAUDE.md). Do not add docstrings.
- **Testing — baseline script:** Use `mash --dry-run --yes <query>` for every intent. Capture only the generated shell command, not full stdout. Cover: move with dest, move without dest, copy, rename same ext, rename different ext, delete file, delete folder, create file with ext, create file without ext, create folder, list, open, cat, chmod.
- **Testing — diff:** Compare generated commands only. `diff test_plans/baseline_output.txt test_plans/refactored_output.txt` must exit 0. If it doesn't, trace the divergence to the specific flow or sanitize step before fixing.
- **Order dependency:** Phase 1 before Phase 2 (intent depends on nothing). Phase 1 + 2 before Phase 3 (Console depends on nothing, but LLMClient imports from `commands/`). Phase 3 + 4 before Phase 5 (flows import all prior modules). Phase 6 after Phase 5. Phase 7 after Phase 6. Phase 8 last.
- **CONTEXT.md:** Phase 0 writes the target architecture into CONTEXT.md before any code changes. Phase 8 verifies it matches reality. This file is the entry point for future Claude sessions — it must accurately describe the module layout, program flow, and responsibilities.
