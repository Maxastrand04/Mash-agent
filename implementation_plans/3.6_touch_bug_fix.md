# 3.6 — Mandatory destination confirmation for create-file / create-folder (Bug 9)

When the user creates a file or folder, mash must always confirm WHERE it lands. Today the destination is prompted for only when the prompt mentions one, and even then the resolved folder is silently dropped from the final `touch` command (`touch ./test.py` instead of `touch ./sandbox/test.py`).

**Goal:** Every create-file and create-folder run goes through one consistent destination flow:

1. If the prompt mentions a destination → resolve candidates, then show a confirmation menu listing them + current dir (`.`).
2. If the prompt does NOT mention a destination → still show a confirmation menu with only `.` as option 1 (plus type-to-search and create-new options). Current dir is the default but is never silently assumed.
3. The chosen destination is then guaranteed to appear in the final command, both for `touch` (create-file) and `mkdir -p` (create-folder). User-visible "Run:" line agrees with what runs.
4. `--yes` and `--dry-run` bypass the confirmation menu per Plan 3.5 Phase 7 contract: auto-select the prompted destination if any, else `.`.

No regression for `mv` / `cp` / rename.

## Claude Instructions

- **The invariant:** create-file and create-folder ALWAYS resolve a destination. There is no code path where `destination` is silently `None` for these intents. The user is asked to confirm via a menu (interactive) or auto-selected to `.` / shallowest match (`--yes` / `--dry-run`).
- **Architecture — why post-processing, not LLM-only:** Plan 3.1 established that menu selections must be honoured even when the LLM ignores hints; `_apply_source` / `_apply_destination` / `_apply_filename` are the contract. Extending `_apply_filename` with an optional `destination` parameter preserves that contract for create-file commands without introducing a new post-processor.
- **Why not extend `_apply_destination` instead:** `_apply_destination` rewrites the last path arg to `dest/`. For `touch foo.py`, that would yield `touch ./sandbox/` — destination only, filename lost. The filename is the user-typed/menu-selected identity and must be preserved; only `_apply_filename` knows it. So the join (`destination + filename`) belongs there.
- **Order:** Phase 1 (templates + system-prompt) → Phase 2 (mandatory destination flow) → Phase 3 (post-processor safety net) → Phase 4 (mkdir prepend). Phase 5 verification depends on all prior phases.
- **Reuse from 3.5:** The `create_destination` flag, the `select_destination` helper, the not-found menu, and the `mkdir -p X && cmd` pattern are already in `run()` for the `mv`/`cp` path. Reuse them — don't fork a parallel implementation. The only differences are: (a) the menu wording adapts for create-intent (b) destination resolution is entered unconditionally for create-file/create-folder, not only when a "to/in" marker is detected.
- **Single-match still confirmed:** Today a single destination match auto-accepts silently. For create-file/create-folder this is changed: even a single candidate is shown as a confirmation prompt (`Use ./<x> as destination?`). For `mv`/`cp` the old single-match auto-accept is preserved — do not regress that.
- **Constraints:** Do not change `_apply_source`, `_apply_destination` (beyond the short-circuit guard in Phase 3), or the rename path. Do not change the format-menu / extension-submenu UX. Do not touch the source-resolution flow. Do not change `mv`/`cp` single-match auto-accept behaviour.
- **Compatibility with Plan 3.5 Phase 7 contract:** Under `--dry-run` / `--yes`, the destination menu is bypassed; auto-select the prompted destination if it resolves, else `.`. The filename format menu also auto-selects format 1. The fix must produce `touch ./<auto-selected-dest>/<auto-selected-name>` under those flags — verify in 5.4 and 5.5.
- **Testing scope:** Every verification command runs inside `Claude_testing/`. Reset fixtures with the same `find ... -not -name "bugs_found.md" -delete` form from test plan 3.0. Never delete `bugs_found.md`.
