# 3.11 — Bug Fixes (confirmed bugs from Test Plan 3.0 run)

Fix the confirmed bugs surfaced by the 3.0 test-plan run, plus the supervisor-clarified follow-ups for 6d.2 and 13.7. Excludes 6a.5 pending a separate decision on `--dry-run` semantics for the list flow.

**Goal:** After this plan:
- Language words in the prompt (`python`, `javascript`, `typescript`, `markdown`, `shell`, etc.) always map to the canonical extension even for single-token bare names.
- Rename preserves `dirname(source)` in every code path (auto-keep, keep-old-ext, use-new-ext, add-ext).
- `--dry-run create a file called <name> in <newdir>` emits `mkdir -p ./<newdir> && touch ./<newdir>/<name>` instead of silently falling back to cwd.
- The source disambiguation menu uses the universal template (header includes `for '<query>':`, footer reads `Select option [1-N], Enter to cancel, type name for other file/folder:` — `Enter` not `N`).
- The list flow under `--dry-run` still prints a trailing `[dry-run] command not executed.` line so the universal flag contract holds.
- The destination-prompt footer wording switches to `type name for other destination:` when used inside a create / move / copy destination prompt (the typed input is interpreted as a destination retype, which IS the canonical behaviour — confirmed by 13.7).
- `--dry-run list …` short-circuits the interactive list flow and prints `Run: ls -la <path>` + `[dry-run] command not executed.`, preserving the universal `--dry-run` contract (fix for 6a.5).
