## Claude Instructions

- **Architecture:**
  - `disambig.py` is the single home for all disambiguation-menu rendering. The `Disambig` class encapsulates the shared `header + numbered options + footer` rendering pattern. Public methods (`pick_from_hits`, `pick_not_found`, `pick_with_actions`) are the only variants — do not inline new menus elsewhere.
  - Each public method returns a tagged `(kind, value)` tuple. Callers branch on `kind`; re-resolution against the directory tree happens at the caller, because only the caller knows whether to search files / dirs / both.
  - `LLMError` is the only exception flow code needs to know about for LLM failures. `LLMClient.ask()` wraps every `ollama.chat()` invocation; everything inside `ask()` either returns a string or raises `LLMError`. `main.py` catches it at the dispatch boundary.

- **Conventions:**
  - Construct `Disambig(console)` in `main.py` alongside `Console` and `LLMClient`, and pass it into flow functions as an extra positional argument. This mirrors the existing `console` / `llm` threading.
  - Tagged kinds use the literals `"selected"`, `"typed"`, `"cancelled"`, `"action"`. Do not introduce synonyms.
  - Use `raise LLMError(...) from e` (not bare `raise LLMError(...)`) so the original ollama traceback is preserved as the `__cause__`.

- **Constraints:**
  - Do **not** keep `pick_from_disambig` as a shim — both callers (in `list.py`) are migrated atomically.
  - Do **not** narrow the `except` in `LLMClient.ask()` to specific ollama exception classes. Catch broad `Exception` and re-raise as `LLMError` — the user-facing message is the same regardless of which underlying failure happened, and enumerating ollama's failure surface is fragile.
  - Do **not** auto-resolve typed input inside `Disambig` methods themselves. The class never touches the directory tree; it is purely a menu renderer + input parser. Re-resolution lives in flow / selection code.
  - Do **not** expand the `deny` list in `.claude/settings.local.json`; only add the two `Bash(...python:*)` entries to `allow`.
  - Do **not** rewrite or refactor existing `mash/intent.py` or `mash/flows/create.py` logic for the "single bare token" behaviour. That behaviour is intentional — Group 1 documents it; it is not changed.
  - Do **not** add a `tree`-first implementation to `files/context`. That is explicitly out of scope and tracked separately.

- **Order dependency:**
  - Phase 0 prerequisites must be confirmed before Phase 1 starts.
  - All four Groups in Phase 1 are independent and can be implemented in parallel.
  - Inside Group 4 only: write the `Disambig` class first, then migrate the call sites — callers will not import successfully until the class exists.
  - The Phase-1 integration `/run-tests` rerun must happen only after all four Groups are ✅.

- **Testing:**
  - Per-Group checks above verify each slice in isolation.
  - The single Phase-1 integration check (a full `/run-tests` rerun) is the AC's final closure criterion (#10) and the only place all four Groups' changes are exercised together.
