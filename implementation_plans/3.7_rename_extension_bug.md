# 3.7 — Rename must preserve file extension unless explicitly changed (Bug 10)

When the user renames a file but does not give an extension in the new name, mash silently drops the original extension (`mv ./test.py ./app` instead of `mv ./test.py ./app.py`). Even when the user gives a different extension, mash should not change it silently without confirming.

**Goal:** The rename flow now enforces extension rules:

1. **New name has no extension** → use the original file's extension verbatim. Never drop it. (`change name of test to app` on `test.py` → `mv ./test.py ./app.py`.)
2. **New name has the same extension as the original** → use as-is. No prompt. (`rename test.py to app.py` → `mv ./test.py ./app.py`.)
3. **New name has a different extension from the original** → prompt the user to confirm the extension change. The user can keep the original extension, accept the new one, or cancel. (`rename test.py to app.txt` → menu.)
4. **Original file has no extension** AND new name has no extension → use the new name as-is, no prompt (nothing to preserve, nothing to change).
5. **Original file has no extension** AND new name has one → prompt the user to confirm adding the extension. Options: keep no extension, accept the new extension, or cancel. Never assume an extension change (or addition) is intentional.

`--yes` and `--dry-run` skip the rule-3 and rule-5 prompts: keep the user-supplied name verbatim under those flags.

## Claude Instructions

- **The invariant:** extensions never change silently. Adding, removing, or replacing an extension always requires explicit confirmation. The only cases where the final extension differs from the original are: (a) rule 3, option 2 confirmed; (b) rule 5, option 2 confirmed; (c) `--yes`/`--dry-run` bypass (user-supplied token wins verbatim).
- **Pure helper + thin caller:** `_reconcile_rename_extension` is pure and easy to unit-test. The interactive confirmation lives in a separate function so the rules and the UX evolve independently.
- **Why blank defaults to "keep original":** safer default. If the user typed a different extension by mistake, the cheap recovery is hitting Enter, not having to type `1`. The menu still shows option 2 as the literal new extension so the user can opt into the change.
- **Order:** Phase 1 (helper) → Phase 2 (prompt) → Phase 3 (wire-in). Phase 4 verification depends on all three. Bug 8 (residual `-f` after retype) from `bugs_found.md` is unrelated and not addressed here.
- **Constraints:** Do not change `_apply_rename`, `_apply_source`, `_apply_destination`, `_apply_filename`. Do not change rename-intent detection (Plan 3.5 Phase 3). Do not touch the move+rename combined flow except to verify it still works (4.11). Do not change the format-menu / extension-submenu used by create-file (that's the new-file naming flow, not rename).
- **Compatibility with Plan 3.5 Phase 7 contract:** `--yes` / `--dry-run` continue to bypass interactive menus. Rule-3 prompt is bypassed under these flags; the user-supplied extension is used verbatim. The extension-preservation rules (1, 2, 4, 5) still apply because they are non-interactive.
- **Testing scope:** Every verification command runs inside `Claude_testing/`. Reset fixtures with `find ... -not -name "bugs_found.md" -delete`. Never delete `bugs_found.md`.
