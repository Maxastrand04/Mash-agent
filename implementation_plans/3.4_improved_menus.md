# 3.4 — Improved disambiguation menus

Let the user type an exact name when no shown choice fits, with a re-check pass through the resolver. Apply this to source, destination, and filename menus, and add single-candidate confirmation for source/destination.

**Goal:** Disambiguation menus never trap the user in the listed options. For source, destination, and new filename selection, the user can always type a name to re-resolve or supply manually. Single-match source/destination prompts for confirmation. Blank input has a sensible default per context (cancel or accept), as specified below.

## Claude Instructions

- **Architecture:** Menu logic lives in `mash/main.py`. `disambiguate()` currently handles only the numbered-list-or-single-match case. Either extend it with an optional `resolver_fn` + `context` to support re-typing, or build per-call inline loops. The source menu is called from the `source_candidates` block, destination from the `dest_candidates` block, and filename from the `new_file` block — all in `run()`.
- **Resolver reuse:** `resolve_paths(context, word)` and `resolve_dirs(context, word)` are pure over the `context` string captured at start of `run()`. Reuse that `context` for re-resolution — do not call `get_directory_context()` again per retype.
- **Blank-input semantics (be explicit, they differ by branch):**
  - Source multi/no-match: blank = cancel.
  - Source single match: blank = confirm; `N` = cancel; other input = retype/re-resolve.
  - Destination multi: blank = cancel.
  - Destination no-match: blank = accept current dir; `N` = cancel.
  - Filename menu: blank = cancel; `1`/`2`/`3` = chosen format; anything else = manual filename.
  - Extension submenu (only if manual filename has no extension and none is known from the prompt): blank = cancel; numeric = listed extension; anything else = manual extension (leading dot optional, stripped before compare) validated against directory extensions ∪ `_EXTENSION_MAP` values. Invalid → re-prompt.
- **Constraints:**
  - Do not change `--yes` / `--dry-run` auto-selection behaviour.
  - Do not change `_apply_source` / `_apply_destination` / `_apply_filename` — only how `resolved`, `destination`, `filename` get populated.
  - Keep prompts short and consistent in tone with existing ones.
- **Order:** Phase 1 → 2 → 3 are independent and can land separately. Phase 4 (refactor) only after the three flows are working, to avoid premature abstraction.
- **Testing:** Use `--dry-run` to drive scenarios end-to-end without touching the filesystem. Run inside `Claude_testing/` per existing convention.
